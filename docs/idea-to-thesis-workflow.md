# Idea 到完整论文初稿工作流

这个工作流用于把一个毕业设计 idea 变成可继续修改的广州商学院本科毕业论文（设计）初稿。目标不是让 Agent 凭空写一篇可以直接提交的论文，而是把“选题、目录、材料缺口、正文初稿、附件材料、排版和审核”组织成可迭代流程。

## 基本原则

- Agent 可以生成论文结构、章节草稿、技术路线、开题报告草稿和 TODO 清单。
- 真实项目、系统功能、实验数据、截图、测试记录和参考文献必须来自用户提供的材料或可核验来源。
- 证据不足的位置必须标记 `TODO`，不能编造用户访谈、实验结果、系统截图、参考文献或不存在的功能。
- 如果还没有代码或系统成果，应先生成“可执行的毕业设计计划”和论文骨架，不要伪装成项目已经完成。
- 最终提交前必须完成 PDF 编译、视觉检查和人工确认。

## 推荐输入格式

```text
选题 idea：

专业/方向：

目标用户：

想解决的问题：

核心功能：
1.
2.
3.

已经有的材料：
- 代码仓库：
- 系统截图：
- 实验/测试数据：
- 参考文献：
- 开题报告：
- 导师要求：
- 学院新增模板或通知：

不能编造的内容：
- 实验数据
- 系统截图
- 用户访谈
- 参考文献
- 已实现功能
```

## 阶段 1：选题判断

Agent 应先判断 idea 是否适合本科毕业论文（设计），并输出：

- 选题是否过大、过小或不适合实现。
- 适合的论文题目候选。
- 推荐研究对象、技术路线和系统边界。
- 需要用户补充的关键事实。

如果 idea 不够具体，先收敛为一个可以实现、可以截图、可以测试、可以写成论文的项目。

## 阶段 2：目录和材料缺口

Agent 生成论文目录时，应同时列出每章需要哪些证据：

```text
第 1 章 绪论
- 可写内容：研究背景、意义、国内外现状、本文工作
- 需要材料：真实参考文献、选题背景、同类系统资料

第 2 章 相关技术
- 可写内容：技术栈、算法或框架介绍
- 需要材料：项目实际使用的技术，不要写未使用技术

第 3 章 需求分析
- 可写内容：用户角色、功能需求、非功能需求
- 需要材料：功能清单、业务流程、用例

第 4 章 系统设计
- 可写内容：架构设计、数据库设计、模块设计
- 需要材料：架构图、ER 图、接口或模块说明

第 5 章 系统实现与测试
- 可写内容：核心功能实现、测试方法、结果分析
- 需要材料：代码、截图、测试记录、真实结果

结论
- 可写内容：完成工作、不足、展望
- 需要材料：项目真实完成情况
```

## 阶段 3：写入模板

确认题目和目录后，Agent 再把内容写入对应文件：

```text
extraTex/meta.tex                 题目、学生信息、关键词
extraTex/front/abstract_zh.tex    中文摘要
extraTex/front/abstract_en.tex    英文摘要
extraTex/body/chapter-*.tex       正文章节
extraTex/back/references.bib      参考文献
extraTex/back/thanks.tex          致谢
extraTex/attachments/*.tex        附件材料
```

只有 idea 时，允许先生成“完整结构 + 可写部分 + TODO”。用户补充材料后，再迭代替换 TODO。

## 阶段 4：参考文献处理

参考文献必须遵守：

- 用户给了真实文献时，优先整理为 `extraTex/back/references.bib`。
- Agent 查找文献时，应记录来源，不要生成无法核验的条目。
- 没有文献来源时，在正文保留 `TODO: 补充真实参考文献`，不要编造 DOI、期刊、作者或年份。
- 编译前检查正文引用是否能和 `.bib` 条目对应。

## 阶段 5：附件材料

毕业论文附件材料册不是可选装饰。Agent 应根据项目阶段补齐：

- 任务书
- 开题报告
- 进展情况记录表
- 答辩记录表
- 评语评分表

如果某些表格需要导师签字、日期或评语，先保留空白或 TODO，不要代替导师编写签字意见。

## 阶段 6：编译和审核

在编译前，让 Agent 做一轮结构、证据和格式自检。用户不需要选择具体能力模块；Agent 会按当前阶段自行调用必要能力。

每次生成或大改后运行：

```bash
python3 scripts/check_structure.py
python3 scripts/doctor.py
```

如果环境可用，再运行：

```bash
bash scripts/build.sh all
```

编译成功后至少检查：

- `main.pdf` 封面、声明、摘要、目录、正文首页、参考文献、致谢。
- `attachments.pdf` 附件封面、目录、任务书、开题报告、进展记录、答辩记录、评分表。
- 页眉、页码、字体、标题、图表编号、参考文献格式和表格是否溢出。

独立审核时，把 `AGENT_REVIEW.md` 交给另一个 Agent。

## 一次性提示词

可以直接复制下面的提示词给 Codex / Claude Code / Cursor：

```text
请使用 gcc-thesis-template 执行 idea-to-thesis 工作流。

我的 idea 是：
【写你的想法】

我的专业/方向是：
【写你的专业或技术方向】

我已经有的材料：
【列出代码、截图、数据、参考文献、导师要求；没有就写暂无】

请按以下顺序工作：
1. 调用本仓库的论文工作流，先判断 idea 是否适合本科毕业论文（设计），给出 2-3 个题目候选。
2. 选择最稳妥的题目，生成论文目录和每章材料缺口。
3. 在不编造事实的前提下，生成完整论文初稿，并把内容写入 extraTex/ 对应文件。
4. 对缺少证据的内容用 TODO 标记，不要编造实验数据、系统截图、用户访谈、参考文献或不存在的功能。
5. 整理附件材料，缺导师签字、日期或评语的位置保留 TODO。
6. 运行 python3 scripts/check_structure.py 和 python3 scripts/doctor.py。
7. 如果环境允许，运行 bash scripts/build.sh all 并检查 main.pdf 和 attachments.pdf。
8. 最后说明已经完成什么、还缺什么真实材料、哪些地方不能确认。
```

## 完成标准

一次 idea-to-thesis 工作流完成后，应至少得到：

- 一个可执行的论文题目。
- 一个符合本科毕业论文结构的目录。
- 一份明确的材料缺口清单。
- 已写入 `extraTex/` 的论文初稿。
- 已整理或标记 TODO 的附件材料。
- 静态结构检查结果。
- LaTeX 环境检查结果。
- 如果环境允许，生成并检查过的 `main.pdf` 和 `attachments.pdf`。
